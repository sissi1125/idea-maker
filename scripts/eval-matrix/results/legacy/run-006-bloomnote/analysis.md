## 核心发现

### 发现 1：T03 再次等同于 T01

Bloomnote PRD 的功能需求章节每个约 340-460 字符，全部小于 chunkSize=512。
markdown-heading-recursive 的递归降级依然不触发，T03 产出的 chunk 边界与 T01 高度一致。

**结论**：在所有已测文档上（PRODUCT.md 3,500 字符 + Bloomnote PRD 12,358 字符），markdown-heading-recursive 的递归降级从未被触发。**该方法的真实效果尚未被评估**，需要章节普遍 > 512 字符的文档才能验证。

### 发现 2：T02 在跨章节综合查询上再次落后（Q5）

Q5（"免费用户和 Pro 用户在功能权益上有哪些区别"）是跨多章节的综合问题：

**T01/T03** 的 top-5 rerank：
```
✓ 0.953  通知与后台产品行为
✓ 0.596  产品功能需求 > 12. 订阅与升级
✓ 0.316  产品概念与业务对象
✗ 0.091  业务规则与校验
✗ 0.004  产品概念与业务对象
```
→ citation 取 top-3：[0.953, 0.596, 0.316]，3/3 超过阈值 0.2 → hitRate=1.00

**T02** 的 top-5 rerank：
```
✓ 0.959  Questions To Confirm（"免费块上限是多少？"待确认事项）
✓ 0.409  外部触点
✗ 0.191  产品功能需求 > 9. 导入、导出与分享
✗ 0.087  跨功能交互流程 > 快速记录每日笔记
✗ 0.071  产品功能需求
```
→ citation 取 top-3：[0.959, 0.409, 0.191]，第三个 0.191 **刚好低于阈值** → hitRate=0.67

T02 的问题：大 chunk（1024 字符）让某些章节包含过多主题，使 cross-encoder 把 "Questions To Confirm"（包含 Pro 权益待确认问题）误判为高度相关，占用了 top 位置；真正关键的"订阅与升级"章节被挤到第三位且分数刚好低于阈值。

### 发现 3：Q4 和 Q6 全部方法均表现差（0.33）

Q4（设计风格与主题色）和 Q6（数据存储与后台行为）对三种方法都是 0.33。
- Q4 的答案分散在多个设计子章节（整体风格、色彩方案），且设计风格描述对 embedding 的语义区分度低
- Q6 的答案跨越"外部触点"（iCloud）和"通知与后台产品行为"两个非功能需求章节，当文档很长时这些章节被埋得更深

这说明存在一类查询——答案分散且语义模糊——**三种 chunk 方法都无法有效召回**。这属于 embedding 层面的限制，chunk 策略无法解决。

| 方法 | Run-005 hitRate (PRODUCT.md) | Run-006 hitRate (Bloomnote PRD) | 提升 |

所有方法在 Bloomnote PRD 上表现更好。T01/T03 提升更大（+0.17），T02 提升较小（+0.11）。

**原因**：Bloomnote PRD 有更多结构化章节（12 个功能章节 + 多个辅助章节），小 chunk 方法（T01/T03）能更精准地定位到具体章节内容，而大 chunk（T02）因章节内容混合而更容易检索到偏题的大块文本。

| 模式 | 结论 | 置信度 |

## 待验证事项

1. **markdown-heading-recursive 的真实价值**：需要章节 > 512 字符的文档（如 API 文档、技术设计文档）
2. **大 chunk 在单章节查询上的优势是否稳定**：Run-006 中 Q1/Q2/Q3 三种方法全部满分，无法区分，需要更难的单章节集中查询来测试
3. **"无法召回"类查询的共性**：Q4/Q6 反映 embedding 层面的语义模糊，与 chunk 无关，后续考虑用 query rewrite 或 summary-keywords transform 改善
