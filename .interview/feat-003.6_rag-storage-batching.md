# feat-003.6 补强：通用 RAG 写库批处理

## 1. 多值 INSERT 为什么能提升 chunk 写入性能？

逐行写入时，每个 chunk 都要经历一次应用到 PostgreSQL 的网络往返、SQL 解析和执行调度。多值 INSERT 把 200 次往返压缩为 4 次（batchSize=50），但仍通过参数传值，不需要牺牲 SQL 注入安全。

## 2. 为什么批量写入还需要事务？

单条多值 INSERT 只能保证当前一批原子。如果 120 行分三批，第三批失败而没有外层事务，前 100 行仍会保留。外层事务让版本选择、旧版本删除和全部批次一起提交或回滚；事务开头再按 documentId 获取 advisory lock，避免两个并发事务读到相同 `MAX(version)`，同时不阻塞不同文档。

## 3. 为什么事务不能直接通过 Pool.query 连续执行？

事务状态属于一条物理数据库连接。`Pool.query()` 的每次调用可能借到不同连接，导致 BEGIN、INSERT、COMMIT 不在同一会话。调用方必须先从 Pool 获取一个 PoolClient，并在事务结束后 release。

## 4. batchSize 为什么限制为 500？

每行使用 13 个 PostgreSQL 参数，500 行是 6500 个参数，明显低于 PostgreSQL 协议参数数量上限，同时避免单条 SQL、参数数组和失败重试范围过大。默认 50 在往返次数与单批故障范围之间取平衡。

## 5. 基准测试为什么使用同一连接、同一临时表并多轮取中位数？

这样能固定连接建立、表结构、向量维度等变量，主要比较 SQL 往返数量；交替测试顺序并取中位数，可减小缓存预热、进程调度和偶发 I/O 抖动的影响。微基准不等于完整 ingestion 端到端加速比。
