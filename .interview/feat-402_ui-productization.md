# feat-402：UI 产品化重构面试题

## 1. 为什么项目总览状态在前端派生，而不是新增后端工作流状态机？

现有资料、Product Brief、Claim 和 Campaign 已经是事实来源。前端只需要根据这些数据判断下一步；再建状态机会产生双重真相和同步问题，也会让纯 UI 改造侵入后端业务模型。

## 2. 这次全栈 E2E 为什么发现了单测没有发现的 ingestion 轮询问题？

后端真实终态是 `succeeded`，前端类型只声明并判断 `completed`。文档表已经变为 ready，因此页面看似成功，但轮询定时器不会停止。只有真实数据库、API 和浏览器一起运行，才能暴露跨层状态契约不一致。

## 3. 如何在不降低可观测性的情况下简化产品？

主流程只展示用户任务和结论，把 evidence、Agent steps、token、成本与评测详情放进按需打开的运行详情。这样透明能力仍然存在，但不会要求新用户先理解 RAG 或 Agent 内部术语。

## 4. 为什么内容工作流状态机放在前端？

Campaign、Variant、异步 job 已经是后端事实。前端纯函数把它们投影成互斥的 `drafting / ready / generating / reviewing / accepted / failed`，能消除散落的 loading 条件，又不会新增需要同步的后端状态。若未来出现跨会话恢复、人工审批 SLA 等持久化需求，再把工作流提升为后端领域模型。

## 5. 为什么视觉资产的卖点关联需要后端字段，而不能只保存在前端？

图片用途和对应卖点属于可复用资产元数据，会被海报、素材筛选和后续生成消费。只存前端会在刷新和跨设备时丢失，也无法做项目隔离校验。因此只增加可空 `claim_id` 和一个受鉴权 PATCH 接口，并在服务端验证 Claim 与资产属于同一项目；这比引入独立素材工作流更小、更可靠。
